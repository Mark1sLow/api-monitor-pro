# Техническое задание на проект: API Monitor Pro

Система мониторинга и анализа SLI/SLO API-методов веб-приложений для DevOps-инженеров.

## 1. Цель проекта

Создание веб-сервиса для мониторинга и анализа ключевых метрик производительности API-методов (SLI/SLO): доступность, задержка ответа, частота ошибок. Система позволяет DevOps-инженерам и командам разработки отслеживать ухудшения эндпоинтов в реальном времени, автоматически рассчитывать статистику производительности, прогнозировать нарушения SLA с использованием машинного обучения, формировать детальные отчеты в различных форматах (PDF, CSV, HTML) и получать своевременные уведомления при нарушении установленных порогов.

## 2. Роли пользователей

- **Гость**: Просмотр общедоступных статистик системы, демонстрационных данных и общих аналитических графиков (если владелец проекта предоставит общий доступ).
- **Зарегистрированный пользователь**: Создание и управление собственными проектами и API-методами, настройка SLI/SLO для каждого эндпоинта, управление расписанием проверок (интервалы от 1 минуты до 1 дня), получение email-уведомлений при нарушениях, просмотр интерактивных графиков метрик, экспорт отчетов в различные форматы, просмотр истории SLA и прогноза нарушений.
- **Администратор**: Полный доступ к управлению всеми пользователями и проектами, просмотр глобальной аналитики и системных отчетов, конфигурация глобальных настроек, просмотр логов системы.

## 3. Модели данных (сущности)

- **Project (Проект)**: Логическая группировка API-методов, относящихся к одному сервису или приложению. Принадлежит одному пользователю.
  - Поля: `name` (название проекта), `url` (базовый URL API), `owner` (ForeignKey на User), `created_at` (дата создания).

- **Endpoint (Эндпоинт)**: Конкретный метод API, который подлежит мониторингу. Привязан к проекту с уникальной комбинацией (project, method, path).
  - Поля: `project` (ForeignKey), `method` (GET, POST, PUT, DELETE), `path` (путь эндпоинта), `sla_latency_ms` (максимально допустимое время ответа в мс), `sla_error_rate` (максимально допустимый процент ошибок, e.g. 0.05 = 5%).

- **Measurement (Измерение)**: Запись результата одной проверки эндпоинта. Содержит все метрики, полученные в процессе однократного HTTP-запроса к API.
  - Поля: `endpoint` (ForeignKey), `timestamp` (время проверки), `response_time_ms` (время ответа в мс), `status_code` (HTTP статус), `is_error` (была ли ошибка), `sla_breached` (был ли нарушен SLA).

- **PerformanceMetrics (Метрики производительности)**: Кэшированная агрегированная статистика за период (час или день) для оптимизации аналитики и графиков. Рассчитывается на основе всех Measurement'ов в период.
  - Поля: `endpoint` (ForeignKey), `period_start`, `period_end`, `response_time_min/max/avg/p50/p95/p99`, `total_requests`, `successful_requests`, `failed_requests`, `error_rate` (%), `sla_breaches`, `sla_compliance` (%), `status_2xx/3xx/4xx/5xx` (распределение статусов).

- **SLATracking (Отслеживание SLA)**: Ежедневная запись о соблюдении или нарушении SLA для каждого эндпоинта. Используется для истории и прогнозирования.
  - Поля: `endpoint` (ForeignKey), `date` (дата отслеживания), `target_sla` (целевой процент SLA), `actual_sla` (фактический процент SLA), `sla_breached` (был ли нарушен SLA), `total_requests`, `breached_requests`, `average_response_time`, `predicted_breach` (прогноз на след. день), `breach_probability` (вероятность нарушения 0-1).

- **Alert (Оповещение)**: История всех уведомлений об нарушениях, отказах и проблемах.
  - Поля: `project` (ForeignKey), `endpoint` (ForeignKey), `alert_type` (sla_breach, endpoint_down, error), `measurement` (связь с конкретным измерением), `message` (текст оповещения), `created_at`, `is_read`.

- **Schedule (Расписание)**: Конфигурация периодичности проверки эндпоинтов для проекта. Один Schedule на Project (OneToOneField).
  - Поля: `project` (OneToOneField), `interval_minutes` (интервал проверки: 1, 5, 10, 15, 30, 60, 360, 1440), `is_active` (включено ли расписание), `last_run`, `next_run` (время последней и следующей проверки).

- **Report (Отчет)**: Сохраненные отчеты в различных форматах (PDF, CSV, HTML) за указанный период.
  - Поля: `project` (ForeignKey), `report_type` (daily, weekly, monthly, custom), `format` (pdf, csv, html), `period_start`, `period_end`, `file_path` (путь к файлу), `total_endpoints`, `total_measurements`, `sla_breaches`.

## 4. Ключевой функционал (User Stories)

* **Создание проекта и регистрация API-методов**: Пользователь проходит регистрацию, создает новый проект, затем добавляет в проект конкретные API-методы с параметрами SLA. Система фиксирует дату создания и связывает проект с владельцем.

* **Автоматическая проверка API и сохранение метрик**: Django Management Command `check_endpoints` запускается по расписанию и выполняет HTTP-запросы ко всем активным эндпоинтам, фиксирует время ответа, статус-код и наличие ошибок. Результаты сохраняются в таблицу Measurement. При нарушении SLA автоматически создается запись Alert.

* **Расчет и кэширование метрик производительности**: Система регулярно агрегирует все Measurement'ы за период, рассчитывает статистику и сохраняет результаты в PerformanceMetrics для быстрого отображения на панели аналитики.

* **Email-уведомления при нарушении SLA**: При нарушении SLA система автоматически отправляет email-уведомление владельцу проекта с деталями нарушения.

* **Интерактивная визуализация графиков и аналитика**: На странице эндпоинта пользователь видит интерактивные графики (Plotly) с тенденциями задержки, доступности и ошибок за последние 7 дней. Красная линия показывает пороги SLA. Таблица содержит последние 10 измерений. Карточки со сводкой показывают количество нарушений за 24 часа, среднюю задержку, текущую доступность.

* **Генерация и экспорт отчетов**: Пользователь может на странице проекта выбрать период (дата начало/конца), тип отчета (дневной, недельный, месячный) и формат (PDF, CSV, HTML), затем система генерирует подробный отчет со всеми метриками, графиками, таблицами и статистикой, и позволяет загрузить файл.

* **Прогноз нарушений SLA (машинное обучение)**: Система анализирует исторические данные SLA за последние 30 дней и использует линейную регрессию (scikit-learn) для предсказания вероятности нарушения в ближайший день. Результаты отображаются на специальной странице "SLA Forecast" с графиком тренда и рекомендациями.

* **Просмотр истории SLA и анализ трендов**: На странице "SLA History" пользователь видит таблицу ежедневных записей SLA, отмечены дни с нарушениями, отображен тренд и вероятность нарушения в следующий день.

* **Dashboard и личный кабинет**: На главной странице залогиненного пользователя отображается:
  - Количество активных проектов и эндпоинтов
  - Количество нарушений SLA за последние 24 часа
  - Список всех проектов с быстрым доступом
  - Последние оповещения

## 5. Внешние интеграции и аналитика

**Интеграции:**
- **SMTP-сервер** для отправки email-уведомлений при нарушении SLA (поддержка Gmail и других провайдеров)
- **HTTP-запросы** (библиотека Requests) к мониторимым API-методам для сбора метрик доступности и производительности
- **Django REST Framework** с JWT-аутентификацией для JSON API (возможность интеграции с внешними системами)
- **Webhooks** для интеграции с Discord, Telegram и другими платформами (Generic webhook)

**Аналитика и обработка данных:**
- **Pandas** для расчета и манипуляции временными рядами, агрегации данных, расчета SLI/SLO статистики
- **NumPy** для высокопроизводительных численных операций и статистических расчетов
- **scikit-learn** (Linear Regression) для машинного обучения и предсказания нарушений SLA на основе исторических трендов
- **Plotly** для создания интерактивных графиков и диаграмм с масштабированием, фильтрацией, экспортом
- **ReportLab** для генерации PDF-отчетов со таблицами, графиками и форматированием
- **Bootstrap 5** для адаптивного дизайна фронтенда

## 6. Изменения в ходе реализации

### 6.1 Архитектура проверки API
- **Планировалось**: Использование Celery + Celery Beat для асинхронных задач и расписания
- **Реализовано**: Django Management Command `check_endpoints` + APScheduler / Cron
- **Причина**: Упрощение архитектуры, лучшая совместимость с хостингом (PythonAnywhere), достаточно для требуемого функционала на текущем этапе, проще деплоить и поддерживать

### 6.2 Расширение моделей и метрик
- **Планировалось**: Базовые метрики (средняя задержка, процент ошибок)
- **Реализовано**: Полный набор перцентилей (P50, P95, P99), min/max, распределение HTTP статусов (2xx/3xx/4xx/5xx), кэшированные PerformanceMetrics, ежедневное SLA tracking
- **Причина**: Обеспечить более глубокую аналитику для DevOps-инженеров, поддержка предсказания и аномалий

### 6.3 Внедрение машинного обучения для прогнозирования
- **Планировалось**: Классическая аналитика с графиками
- **Реализовано**: Прогноз нарушений SLA на основе Linear Regression, анализ трендов, обнаружение аномалий
- **Причина**: Предоставить DevOps-командам инструменты для проактивного выявления проблем

### 6.4 Система уведомлений
- **Планировалось**: Только Email
- **Реализовано**: Email, Alert history с отметкой о прочитанности, интеграция с DRF для webhook'ов (Discord, Telegram, Generic)
- **Причина**: Разные команды используют разные каналы коммуникации, webhooks обеспечивают гибкость интеграции

### 6.5 Система экспорта отчетов
- **Планировалось**: Базовый CSV экспорт
- **Реализовано**: Три формата (PDF с таблицами и графиками, CSV для аналитики, HTML для просмотра), шаблонные отчеты (дневной, недельный, месячный, кастомный), система Report с сохранением истории
- **Причина**: Разные требования у разных стейкхолдеров

### 6.6 API для интеграции
- **Планировалось**: Только веб-интерфейс
- **Реализовано**: Django REST Framework с JSON endpoints для метрик, SLA forecast, CSV export; JWT-аутентификация для безопасности
- **Причина**: Возможность интеграции с другими системами мониторинга
