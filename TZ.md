# Техническое задание на проект: API Monitor Pro

Система мониторинга и анализа SLI/SLO API-методов веб-приложений для DevOps-инженеров.

## 1. Цель проекта

Создание веб-сервиса для мониторинга и анализа ключевых метрик производительности API-методов (SLI/SLO): доступность, задержка ответа, частота ошибок. Система позволяет DevOps-инженерам и командам разработки отслеживать ухудшения эндпоинтов в реальном времени, автоматически рассчитывать статистику производительности, прогнозировать нарушения SLA с использованием машинного обучения, формировать детальные отчеты в различных форматах (PDF, CSV, HTML) и получать своевременные уведомления при нарушении установленных порогов.

## 2. Роли пользователей

- **Зарегистрированный пользователь**: Создание и управление собственными проектами и API-методами, настройка SLI/SLO для каждого эндпоинта, управление расписанием проверок, получение email-уведомлений при нарушениях, просмотр интерактивных графиков метрик, экспорт отчетов в различные форматы, просмотр истории SLA и прогноза нарушений.
- **Администратор**: Полный доступ к управлению всеми пользователями и проектами, просмотр глобальной аналитики и системных отчетов, конфигурация глобальных настроек, просмотр логов системы.

## 3. Модели данных (сущности)

- **Project (Проект)**: Логическая группировка API-методов, относящихся к одному сервису или приложению.
- **Endpoint (Эндпоинт)**: Конкретный метод API, который подлежит мониторингу.
- **Measurement (Измерение)**: Запись результата одной проверки эндпоинта. Содержит все метрики, полученные в процессе однократного HTTP-запроса к API.
- **PerformanceMetrics (Метрики производительности)**: Кэшированная агрегированная статистика за период для оптимизации аналитики и графиков. Рассчитывается на основе всех Measurement'ов в период.
- **SLATracking (Отслеживание SLA)**: Ежедневная запись о соблюдении или нарушении SLA для каждого эндпоинта. Используется для истории и прогнозирования.
- **Alert (Оповещение)**: История всех уведомлений об нарушениях, отказах и проблемах.
- **Schedule (Расписание)**: Конфигурация периодичности проверки эндпоинтов для проекта. 
- **Report (Отчет)**: Сохраненные отчеты в различных форматах (PDF, CSV, HTML) за указанный период.

## 4. Ключевой функционал (User Stories)

* **Создание проекта и регистрация API-методов**: Пользователь проходит регистрацию, создает новый проект, затем добавляет в проект конкретные API-методы с параметрами SLA. Система фиксирует дату создания и связывает проект с владельцем.

* **Автоматическая проверка API и сохранение метрик**: Django Management Command `check_endpoints` запускается по расписанию и выполняет HTTP-запросы ко всем активным эндпоинтам, фиксирует время ответа, статус-код и наличие ошибок. Результаты сохраняются в таблицу Measurement. При нарушении SLA автоматически создается запись Alert.

* **Расчет и кэширование метрик производительности**: Система регулярно агрегирует все Measurement'ы за период, рассчитывает статистику и сохраняет результаты в PerformanceMetrics для быстрого отображения на панели аналитики.

* **Email-уведомления при нарушении SLA**: При нарушении SLA система автоматически отправляет email-уведомление владельцу проекта с деталями нарушения.

* **Интерактивная визуализация графиков и аналитика**: На странице эндпоинта пользователь видит интерактивные графики с тенденциями задержки, доступности и ошибок за последние 7 дней. Красная линия показывает пороги SLA. Таблица содержит последние 10 измерений. Карточки со сводкой показывают количество нарушений за 24 часа, среднюю задержку, текущую доступность.

* **Генерация и экспорт отчетов**: Пользователь может на странице проекта выбрать период, тип отчета и формат, затем система генерирует подробный отчет со всеми метриками, графиками, таблицами и статистикой, и позволяет загрузить файл.

* **Прогноз нарушений SLA**: Система анализирует исторические данные SLA за последние 30 дней и использует линейную регрессию для предсказания вероятности нарушения в ближайший день. Результаты отображаются на специальной странице "SLA Forecast" с графиком тренда и рекомендациями.

* **Просмотр истории SLA и анализ трендов**: На странице "SLA History" пользователь видит таблицу ежедневных записей SLA, отмечены дни с нарушениями, отображен тренд и вероятность нарушения в следующий день.

* **Личный кабинет**:
  - Количество активных проектов и эндпоинтов
  - Количество нарушений SLA за последние 24 часа
  - Список всех проектов с быстрым доступом
  - Последние оповещения

## 5. Внешние интеграции и аналитика

**Интеграции:**
- **SMTP-сервер** для отправки email-уведомлений при нарушении SLA
- **HTTP-запросы** (библиотека Requests) к мониторимым API-методам для сбора метрик доступности и производительности
- **Django REST Framework** с JWT-аутентификацией для JSON API

**Аналитика и обработка данных:**
- **Pandas** для расчета и манипуляции временными рядами, агрегации данных, расчета SLI/SLO статистики
- **NumPy** для высокопроизводительных численных операций и статистических расчетов
- **scikit-learn** для предсказания нарушений SLA на основе исторических трендов
- **Plotly** для создания интерактивных графиков и диаграмм с масштабированием, фильтрацией, экспортом
- **ReportLab** для генерации PDF-отчетов со таблицами, графиками и форматированием
- **Bootstrap 5** для адаптивного дизайна фронтенда

## 6. Изменения в ходе реализации

### Архитектура проверки API
- **Планировалось**: Использование Celery + Celery Beat для асинхронных задач и расписания
- **Реализовано**: Django Management Command `check_endpoints` + APScheduler / Cron
- **Причина**: Упрощение архитектуры, лучшая совместимость с хостингом (PythonAnywhere), достаточно для требуемого функционала на текущем этапе, проще деплоить и поддерживать
